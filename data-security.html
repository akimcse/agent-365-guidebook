데이터 보안 · Purview

에이전트가 다루는 데이터를 Microsoft Purview로 어떻게 분류·보호·감사하는지 정리합니다. 실제 분류·탐지 엔진(SIT·Trainable Classifier·EDM)과 이를 소비하는 정책(DLP·Communication Compliance·IRM), 이를 집계해 보여주는 DSPM을 계층으로 구분하고, 어디까지가 Purview 고유 기능이고 어디서부터 Agent 365가 필요한지를 명확히 합니다.

이 문서를 읽는 핵심 렌즈 — 소유 경계Purview 데이터 보안의 대부분은 Purview 고유 기능이며, M365 Copilot 상호작용은 per-user 라이선스로 커버됩니다(이것이 기본값). Copilot Studio(비 Teams 채널)·Foundry·외부 AI 등 非 M365 AI로 확장하려면 PAYG 청구가 필요합니다. Agent 365는 데이터 통제를 새로 추가하지 않고 DSPM AI observability 인벤토리에 에이전트를 표출합니다(런타임 탐지는 문서④ Defender).
태그 규칙 — 기본(Purview·M365)은 태그 없음. 예외만 표시: PAYG 非M365 확장 · E5 상위 라이선스 · Preview 미리보기 · Agent 365 인벤토리 표출.
SECTION 01

데이터 보안 아키텍처분류 엔진 → 라벨링 → 정책 집행 → DSPM 집계 · Audit/eDiscovery 포렌식

🔑 데이터 출처와 계층: 모든 데이터는 에이전트 상호작용(프롬프트·응답·접근 리소스)에서 시작합니다. 분류 엔진 SIT와 Trainable Classifier는 별개 엔진으로 각각 민감 데이터를 탐지하고, 라벨링이 라벨을 적용하며, 정책(DLP·CC·IRM)이 이를 소비해 차단·탐지하고 그 신호가 DSPM으로 집계됩니다. 한편 Audit·eDiscovery는 정책과 무관하게 상호작용을 직접 기록합니다(감사가 켜져 있으면 자동) — 그래서 DSPM은 이 감사 데이터까지 함께 끌어와 보여줍니다. DSPM 자체는 분류 엔진이 아닙니다.
