런타임 위협 탐지

Copilot Studio / Azure AI Foundry 에이전트의 런타임 위협 탐지·차단은 서로 다른 3개 계층에서 일어납니다 — ① Copilot Studio 네이티브, ② Prompt Shields → Defender for AI(모델 레벨), ③ Agent 365 · Defender XDR(에이전트 런타임). 계층마다 탐지 엔진 · 커넥터 · 시그널 흐름 · 표출 포털 · 라이선스가 다릅니다.

SECTION 01

3계층 탐지 아키텍처시그널 소스 → 탐지 엔진 → 커넥터 게이트 → 표출 포털

🎯 핵심 "Agent 365로 jailbreak 탐지" 자체는 Azure 구독 없이 Layer ① + ③으로 가능합니다. 모델 수준(Azure OpenAI/Foundry)의 Defender for AI 탐지(Layer ②)까지 원하면 Azure 구독 + MDC 플랜이 필요합니다. 중요: Layer ②에서 실제 탐지·차단은 Prompt Shields(Azure AI Content Safety)가 하고, Defender for AI는 그 결과를 alert로 표출하는 별개 서비스입니다 — 하나로 합쳐 이해하면 안 됩니다.
⚠️ 2026.7.1 라이선스 이관 (검증됨) 에이전트 레벨(Copilot Studio·Foundry 에이전트) 탐지·실시간 보호는 Defender for Cloud Apps/Defender for Cloud → Agent 365 라이선스로 이관(해당 라이선스 없으면 접근 상실). 단 모델 레벨(Azure OpenAI 등 Foundry Models) jailbreak 탐지는 Defender for AI Services(Azure 구독)에 잔류. 이관일에 기존 Block 규칙이 중단되므로 새 real-time protection 정책으로 재정의 필요.

SECTION 03

Defender 포털"Security for AI" 설정의 커넥터가 어떻게 엮여 합쳐지나

Defender 포털 System › Settings › Security for AI 화면은 두 그룹으로 나뉘며, 각 그룹에 에이전트 유형별 커넥터가 있습니다. 이 커넥터들이 SECTION 01 아키텍처의 posture(사전 가시성·조사)와 real-time protection(런타임 탐지·차단)을 각각 켜고, 마지막 M365 App 커넥터가 Agent 365 observability 로그를 수집해 조사·Advanced Hunting을 묶어 모두 Defender 포털에서 합쳐집니다.

Microsoft Defender 포털 (XDR)

posture(인벤토리·attack path·exposure) + real-time protection(Incidents & Alerts) + M365 커넥터(조사·Advanced Hunting)가 한 포털로 수렴

🔑 M365 App 커넥터 = linchpin. 이게 연결돼야 RTP 탐지·차단의 alert·조사·Advanced Hunting이 Defender 포털에 모입니다. 커넥터를 켜면 Agent 365 observability 로그(CS·Foundry 에이전트 신호)가 수집됩니다. 끊기면 SECTION 01 아키텍처대로 차단은 작동하지만 alert·조사는 포털에 미표출됩니다. (2026.7.1부로 에이전트 보안은 기존 Defender for Cloud Apps 경로에서 Agent 365 라이선스로 이관됨)
