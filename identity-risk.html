에이전트 신원 · 위험

Copilot Studio 에이전트가 어떤 신원으로 인증하고, 그 sign-in이 어떤 로그에 찍히며, 어떤 위험 리포트(Risky Users vs Risky Agents)로 라우팅되는지 정리합니다.

SECTION 04

실측 + 문서로 확정된 사실 (실측 데모 테넌트 · 문서 Microsoft Learn)

1 커넥터 툴 = 항상 User 실측문서

SharePoint·Teams 툴 호출은 End-user든 Maker든 전부 User sign-in에 찍힘. 데모에서 "전부 User로 찍힘" 확인. 에이전트 신원 sign-in 없음.

2 Teams 채널 대화 = 에이전트 신원 실측문서

Contoso Helper에 Teams로 말 걸자 Service principal 탭에 SP sign-in 실제로 찍힘 (SP ID e10678ab…). 채널만이 Entra Agent ID로 E2E 인증.

3 메시지 수 ≠ sign-in 수 실측

같은 세션에서 연속 메시지 보내도 토큰 캐싱(~60–90분)으로 로그 안 늘어남. 간격 두고 새 상호작용하면 새 sign-in 발생. → 연료는 "새 세션/토큰 발급".

4 Digital Worker = Risky Agents 문서

OBO 아님. agentUser는 riskyAgents의 하위타입 riskyAgentUser로 라우팅. CA도 "Agent risk" 조건 사용. User가 아니라 Agent 위험.

SECTION 05

Risky Agents 탐지 목록모두 "에이전트 자체 신원 활동"(SP / agentUser) 기반

signInSpike에이전트 sign-in 볼륨 비정상 급증 ← 우리 표적
unfamiliarResourceAccess평소 안 쓰던 리소스 접근
entraDirectoryReconnaissance디렉터리 정찰성 조작
failedAccessAttempt무권한 리소스 접근 실패
earlyLifeMaliciousActivity신규 에이전트의 이상 행동
threatIntelligenceAccount알려진 공격 패턴
suspiciousCredentialUsage새 자격증명 추가 후 사용
riskyAgentIdentityBlueprintPrincipal블루프린트 주체 위험
🔑 signInSpike 유발 공식: 표면 = Teams(유일하게 Agent ID 인증) · 연료 = 새 세션/토큰 발급(메시지 수 아님) · 방법 = 다수 사용자 × 새 대화 또는 DirectLine API로 conversation 대량 생성 · 관측 = Service principal 탭 · 검증 = Risky agents (ML 배치, 최대 24h 지연).
⚠️ 문서가 침묵하는 갭 (확정 전 주의)
  1. 개별 탐지(signInSpike 등)가 agentUser에도 모두 적용되는지는 타입별로 명시 없음 (같이 적용될 가능성 높으나 미확정).
  2. riskyAgentUser 위험이 riskyUsers에도 동시 게시되는지 명시 없음 (CA 설계상 riskyAgents 전용이 강하게 시사됨).
  3. M365 Copilot 표면이 SP sign-in을 생성하는지는 격리 실측 미완 — "for Web" 로그가 반례일 수 있어 확정 문구는 실측 후.
  4. 관련 Graph API 대부분 /beta (agentUser는 v1.0).
