에이전트 유형 분류

실행 위치(SaaS/Endpoint/Cloud)·역할(Productivity/Action/Automation)·제작 방식(Pro-code/Low-code)의 3축으로 AI 에이전트를 분류합니다. 모든 정의는 Microsoft Learn 공식 문서 기반입니다.

AXIS 01

실행 위치 기반: SaaS · Endpoint · Cloud

Microsoft는 에이전트가 “앱(SaaS)·엔드포인트·클라우드에 걸쳐(span apps, endpoints and cloud)” 확산된다고 설명하며, 이 셋을 서로 다른 실행 경계로 다룹니다.[1]

🧭
거버넌스 관점 — SaaS는 커넥터·데이터 경계, Endpoint는 엔드포인트·런타임 경계, Cloud는 신원·오케스트레이션 경계가 핵심 통제 지점입니다. Microsoft Defender는 실제로 “클라우드 에이전트” 탭과 “로컬 에이전트” 탭으로 이 둘을 분리해 관리합니다.[4]
Type 01

SaaS

즉시 사용형(ready-to-use) 파트너 에이전트
실행 위치SaaS 공급자의 클라우드 컨트롤 플레인 또는 SaaS 테넌트 안에서 원격 실행
접근 방식OAuth·커넥터·카탈로그 통합을 기반으로 벤더가 API를 호출하거나 웹에서 작업 수행
예시Box·Canva·Figma·Zendesk·Zoho 등 사전 통합 파트너 에이전트
주요 위험
  • 민감 데이터 노출
  • 과다 권한 커넥터
  • 섀도 온보딩 · 가시성 사각
  • 업무 앱 전반의 도구 오용
Type 02

Endpoint

터미널 · IDE · 데스크톱 앱 · 온디바이스 런타임
실행 위치Windows·macOS 엔드포인트에서 사용자 권한(user-level permissions)으로 디바이스 위에서 직접 실행[3]
접근 방식파일·셸·브라우저·로컬 시크릿·MCP 서버·리포지토리 컨텍스트에 직접 접근. Defender가 온보딩된 디바이스에서 자동 검색
예시Claude Code · Codex CLI · Cursor · ChatGPT Desktop · Ollama Desktop 등 (Defender 지원 목록 기준)[3]
주요 위험
  • 로컬 콘텐츠발(發) 프롬프트 인젝션
  • 시크릿 · 소스코드 유출
  • 디바이스 상 명령 오용
  • 일반 프로세스로 취급되어 가시성 부재
Type 03

Cloud

관리형 에이전트 서비스 또는 Cloud PC
실행 위치클라우드 호스팅 에이전트 서비스 또는 관리형 Cloud PC에서 분리된 세션으로 실행
접근 방식관리 ID(managed identity), REST·SDK·API, 워크플로 트리거, 원격 앱·브라우저 세션
예시Microsoft Foundry Agent Service, Amazon Bedrock · Google Vertex AI 에이전트, Windows 365 for Agents[4]
주요 위험
  • 대규모 도구·API 오용
  • 워크플로 전반의 권한 드리프트
  • 메모리 포이즈닝 · 프롬프트 공격
  • 에이전트 스프롤 · 오케스트레이션 실패
🎯
핵심 요지 — 같은 ‘AI 에이전트’라도 SaaS · Endpoint · Cloud는 서로 다른 실행 경계와 공격면을 가집니다. 특히 Endpoint는 사용자 권한으로 일반 프로세스처럼 실행되기 때문에 엔드포인트 텔레메트리와 런타임 제어가 가장 중요합니다.[3]

Endpoint 세부 계층

Microsoft Defender는 로컬 에이전트를 엔드포인트 자산으로 취급해 CLI · IDE · 데스크톱 · 런타임 단위로 검색·관리합니다.[3]

계층실행 / 접근예시핵심 리스크
Coding CLI터미널터미널 · 리포 · 셸 · 로컬 시크릿GitHub Copilot CLI · Codex CLI · Claude Code · Gemini CLI명령 실행, 파일 쓰기, 시크릿 유출
IDE · 확장워크스페이스IDE 워크스페이스 · 확장 · 터미널 · gitCursor · Windsurf · VS Code(Cline / GitHub Copilot)워크스페이스 포이즈닝, 과도한 리포 범위
Desktop AI앱데스크톱 앱 · 파일 첨부 · 사용자 세션ChatGPT Desktop · Claude Desktop · Devin Desktop열린 콘텐츠 인젝션, 클립보드·파일 유출
AI Runtime온디바이스온디바이스 런타임 · 로컬 API·캐시 · 자율 플랫폼OpenClaw · Nanobot · Ollama Desktop불투명한 로컬 실행, 관리되지 않는 도구·모델 체인

AXIS 02

역할 기반: Productivity · Action · Automation

Microsoft Cloud Adoption Framework는 “조직이 일반적으로 배포하는 세 가지 에이전트 범주”를 정의하며, 각 범주는 서로 다른 자율성(autonomy) 수준을 가집니다.[6]

낮은 자율성 (단순) 높은 자율성 (고급)
Type 01

Productivity

예: Microsoft 365 Copilot Chat
정의(번역)정보 검색과 종합에 초점을 맞춰 의사결정을 가속합니다. 지식 도구(knowledge tools)로 여러 소스에서 데이터를 끌어와 사용자에게 제공합니다.[6]
대표 시나리오고객 서비스 지원 · 사내 지식 관리
실무 판단 사람이 정보를 더 빠르게 이해·활용하도록 돕는 것이 주 목적일 때 적합합니다.
Type 02

Action

예: Copilot Studio 에이전트
정의(번역)정해진 워크플로 안에서 레코드 업데이트·프로세스 트리거 같은 특정 작업을 수행합니다. 지식 도구와 실행 도구(action tools)를 결합해 작업을 완수합니다.[6]
대표 시나리오서비스 티켓 생성 · 시스템 모니터링
실무 판단 단순 응답을 넘어 범위가 정해진 업무 액션까지 완결해야 할 때 적합합니다.
Type 03

Automation

예: Microsoft Scout
정의(번역)복잡한 다단계 프로세스를 최소한의 감독으로 관리합니다. 지식 도구·실행 도구에 더해, 언제 실행·중단·에스컬레이션할지 결정하는 트리거를 사용합니다.[6]
대표 시나리오공급망 최적화
실무 판단 프로세스 자율성이 가장 중요할 때 적합하지만, 더 강한 거버넌스가 필요합니다.
🎯
핵심 요지 — 이 3분류는 단순 기능 목록이 아니라, 지식 지원(knowledge support) → 워크플로 실행 → 프로세스 자율성으로 올라가는 엔터프라이즈 설계 스펙트럼입니다. CAF 기술 계획 문서에서는 동일한 자율성 축을 Retrieval · Task · Autonomous 라벨로도 표기합니다.[7]
근거 · [6] CAF — AI agents 개요(Productivity/Action/Automation 정의) · [7] CAF — 기술 솔루션 계획(Retrieval/Task/Autonomous)
AXIS 03

제작 방식 기반: Pro-code · Low-code

사용자가 직접 만드는 커스텀 에이전트를 “누가, 어떤 표면에서 만드느냐”로 구분합니다. CAF 기술 계획 문서는 “Copilot Studio의 low-code 에이전트와 Microsoft Foundry의 pro-code 솔루션을 비교하라”고 명시적으로 두 방식을 대비합니다.[7]

🧭
거버넌스 관점 — Pro-code는 코드·SDK 통제(파이프라인·아티팩트·의존성), Low-code는 플랫폼·커넥터 통제(테넌트·라이선스·거버넌스 정책)가 핵심 지점입니다.
Type 01

Pro-code

개발자가 SDK·프레임워크·코드로 오케스트레이션·도구·메모리를 직접 구현
제작 주체소프트웨어 엔지니어 · ML·플랫폼 팀
제작 표면IDE · Git 리포 · CI/CD 파이프라인 · 클라우드 SDK
대표 예시Microsoft Foundry Agent Service(hosted agents) · Microsoft Agent Framework · LangGraph · OpenAI Agents SDK[8] · Google Vertex AI ADK[9] · Amazon Bedrock AgentCore[10]
자유도 / 속도자유도 매우 높음(임의 로직·멀티에이전트·모델 세밀 제어) · 대신 엔지니어링 투자·유지보수 부담이 큼[11]
주요 위험
  • 취약 의존성 · 공급망(supply-chain) 리스크
  • 하드코딩 시크릿 · 과도한 IAM 권한
  • 커스텀 도구 코드의 검증 부재
  • 섀도 AI(개인 리포·API 키)
Type 02

Low-code

비개발자(현업·메이커)가 GUI·자연어·커넥터로 조립
제작 주체현업 담당자 · 시티즌 디벨로퍼 · 솔루션 컨설턴트
제작 표면웹 스튜디오 · 드래그앤드롭 캔버스 · 프롬프트·자연어 설정
대표 예시Microsoft Copilot Studio[12] · Salesforce Agentforce[13] · (업계 통용) Palantir AIP · ServiceNow AI Agents · UiPath Agent Builder
자유도 / 속도플랫폼 제공 커넥터·템플릿·가드레일 범위 내(중간 자유도) · 대신 수 시간~일 단위로 빠른 프로토타이핑[7]
주요 위험
  • 커넥터 과다권한 · 데이터 경계 침범
  • 시티즌 개발자 거버넌스 사각(섀도 온보딩)
  • 플랫폼 락인 · 불투명한 내부 동작
  • DLP·감사 로그가 플랫폼 능력에 종속

Microsoft 공식 대비 표 (CAF 기술 솔루션 계획)

CAF는 솔루션별 제작 접근 방식을 아래와 같이 명시합니다.[7]

솔루션제작 접근 방식(Approach)지원 에이전트 유형
SaaS agents즉시 사용형 (SaaS)Retrieval · Task
Microsoft FoundryPro-code 및 low/no-code (PaaS)Retrieval · Task · Autonomous
Microsoft Copilot StudioLow/no-code (SaaS)Retrieval · Task · Autonomous
GPU · 컨테이너Pro-code (PaaS 또는 IaaS)Retrieval · Task · Autonomous
⚠️ 인용 정확성 주석 — Google Vertex AI ADK·Amazon Bedrock(Pro-code), Microsoft Copilot Studio·Salesforce Agentforce(Low-code)는 공식 문서로 확인했습니다. Palantir AIP · ServiceNow AI Agents · UiPath Agent Builder는 업계에서 통상 low-code로 분류되나 이번 검토에서 공식 1차 출처를 확보하지 못해 ‘업계 통용 예시’로만 표기했습니다(공식 인용 없음).
🎯
핵심 요지 — Pro-code는 자유도↑ · 거버넌스 책임을 조직이 직접 진다, Low-code는 속도↑ · 거버넌스를 플랫폼에 위임한다. 통제 지점이 근본적으로 다르므로 감사·DLP·권한 관리 방식도 트랙을 분리해 설계해야 합니다.
