✦ 심화 문서 ⑤

거버넌스 프레임워크

여러 플랫폼(Copilot · Foundry · Vertex AI · Bedrock)에 흩어진 에이전트를 하나의 거버넌스 컨트롤 플레인으로 관리하는 아키텍처를 제시합니다. 한 번의 에이전트 액션이 요청 → 게이트 → 실행 → 관측으로 흐르는 런타임 경로를 기준으로, Access & Identity · Policy · Controls · Observability 네 축이 실제로 어떤 Microsoft 보안 솔루션과 Agent 365 역할로 동작하는지를 앞선 문서 ②③④와 엮어 설명하고, 마지막에 전체를 한 판으로 매핑합니다.

표기 원칙SECTION 02·03의 게이트별 상세는 앞선 심화 문서와 이어집니다 — 각 셀의 (문서 N ↗) 링크로 해당 문서로 바로 이동할 수 있습니다.
SECTION 01

거버넌스 아키텍처 개요한 번의 액션이 요청 → 게이트 → 실행 → 관측으로 흐르는 경로

🔑 읽는 법: 에이전트가 액션(prompt·tool 호출·A2A)을 내면 AI Gateway(정책 집행 지점)가 실행 전에 가로채 세 게이트 — ① Access & Identity(누가?) → ② Policy(허용되나?) → ③ Controls(허용·차단·플래그) — 를 통과시킵니다. ALLOW일 때만 실행되고, 모든 단계의 메타데이터·텔레메트리가 중앙 레지스트리로 흘러 감사·근본원인·오판(orphan)·위험 탐지를 가능하게 합니다.
SECTION 02

각 게이트는 실제로 어떻게 동작하나3 필라 → Microsoft 보안 솔루션 · Agent 365 역할 · 실제 액션

게이트 ① Access & Identity — "누가 요청하나?"  Entra 관리자
신원 부여Microsoft Entra Agent ID — 에이전트마다 고유 agent identity, blueprint로 다수 에이전트에 CA·거버넌스·감사를 일괄 적용
접근 제어Conditional Access for agents — 실시간 agent identity risk 신호로 접근 제어. Microsoft Managed Policies가 high-risk 에이전트를 기본 차단
위험 탐지Identity Protection for agents — 사용자 위험에서 유래한 위험 + 에이전트 자체 행동 기반 이상/비인가 활동 탐지
집행 지점AI gateway (Azure API Management) — 액션을 실행 전 가로채 신원 부착·토큰 발급·쿼터. OpenAI/Anthropic/Vertex/Bedrock 백엔드까지 관리
💡 실제 액션 순서: AI gateway가 액션을 intercept → 에이전트 신원 부착 → CA가 agent identity risk를 평가 → high-risk면 차단, 아니면 유효 권한(OBO/앱 전용)을 해석해 다음 게이트로. (문서 ② ↗)
출처 entra/security-for-ai-overview (CA·Identity Protection) · entra-agent-id(blueprint·멀티플랫폼) · apim/genai-gateway
게이트 ② Policy — "이 액션이 허용되나?"  Purview · Defender 관리자

액션의 콘텐츠(모델 입출력)와 데이터(민감정보)를 각각 다른 엔진이 평가합니다.

콘텐츠 정책Azure AI Content Safety — Prompt Shields — jailbreak·XPIA를 모델 입출력 단에서 탐지·차단. Defender for Cloud AI threat protection이 이를 XDR 알림으로 표출 (문서 ④ Layer② ↗)
데이터 정책Purview — SIT·EDM으로 민감정보 분류, Sensitivity Labels(EXTRACT 게이트)로 접근 통제, DLP가 민감 데이터의 Copilot 처리를 차단 (문서 ③ ↗)
위험 점수화Purview Insider Risk Management(Risky AI usage · Risky Agents) — 위험 프롬프트·민감 응답·외부 공유를 점수화해 Adaptive Protection으로 연결
💡 실제 액션 순서: 프롬프트·응답·접근 리소스를 분류 엔진이 평가 → 콘텐츠 위협은 Prompt Shields가, 민감 데이터는 Purview DLP가 정책 매칭 → 위반 시 차단·플래그. (문서 ③ ↗) (문서 ④ ↗)
출처 content-safety/prompt-shields · defender-for-cloud/ai-threat-protection · purview/ai-agent-365 (DLP·IRM·라벨)
게이트 ③ Controls — "허용 · 차단 · 플래그"  Defender 관리자
런타임 규칙Microsoft Defender — Security for AI 실시간 보호. Default 규칙(audit): 차단 없이 BehaviorInfo 기록 · Custom 규칙(block): 실행 전 tool 호출 차단 (문서 ④ Layer③ ↗)
차단 4종Secret exfiltration · Malicious content propagation · Evasion techniques · Unsafe email domain
결과 표출audit 모드 → XDR Alert · block 규칙 → BehaviorInfo 기록. 모든 결과가 Microsoft Defender 포털(XDR)로 수렴해 조사·상관
데이터 차단Purview DLP가 라벨·SIT 조건으로 민감 데이터 처리 자체를 사전 차단(보완적 통제)
💡 실제 액션 순서: tool invocation이 tool 경계에 도달 → 규칙 평가 → ALLOW면 실행, 위협이면 차단·플래그 → XDR로 표출. audit 모드에서만 알림, block 규칙이 덮으면 BehaviorInfo만 기록됩니다. (문서 ④ ↗)
출처 real-time-agent-protection (audit/block 규칙) · security-for-ai(XDR 표출)
Agent 365 — 4개 관리자 역할이 컨트롤 플레인을 나눠 맡습니다Entra 관리자(신원·CA·거버넌스) · Defender 관리자(위협 탐지·실시간 통제) · Purview 관리자(데이터 분류·DLP·컴플라이언스) · Microsoft 365 관리자(레지스트리·온보딩·수명주기). Agent 365가 이 넷을 하나의 Agent registry로 묶어 Observe · Govern · Secure를 제공합니다.
SECTION 03

Observability — 중앙 Agent Registry정규화된 에이전트 레코드 · Agent 365 레지스트리 매핑

모든 게이트·실행 단계의 메타데이터·텔레메트리가 중앙 레지스트리로 흘러 정규화된 레코드로 집계됩니다 — 전 플랫폼 에이전트 인벤토리·소유권, 액션·차단 감사, 근본원인·오판(orphan)·위험 탐지의 근거가 됩니다. Observability는 단일 제품이 아니라 여러 Microsoft 솔루션의 관측 데이터가 수렴하는 계층입니다.

어느 Microsoft 솔루션에 매핑되나
인벤토리·소유권 주 매핑Microsoft 365 admin center — Agent registry(Agent 365) — 전 에이전트 단일 뷰 · owner 없음 · unmanaged(shadow) · at-risk 집계
데이터 관측Purview DSPM AI observability — oversharing · exfiltration · 이상 접근 추적 (문서 ③ ↗)
위협 관측Microsoft Defender XDR — alert · incident · BehaviorInfo 상관·조사 (문서 ④ ↗)
신원 관측Entra 인벤토리 — agent identity 중앙 발견·관리 · shadow AI 방지 (문서 ② ↗)

Microsoft Agent 365 레지스트리 매핑: M365 admin center의 Agent registry가 동일 역할 — 전 에이전트를 단일 뷰로 관리하고 owner 없음 · unmanaged(shadow) · at-risk를 집계합니다. 위험 신호는 Entra·Defender·Purview·M365 admin에서 유입됩니다: Shadow agent · No owner · Excessive permissions(Critical) · Prompt injection · Sensitive data access · Conditional access violation(High) 등. DSPM AI observability가 oversharing·exfiltration·이상 접근을 추적합니다.

SECTION 04

전체 매핑 아키텍처 — 한 판거버넌스 게이트 × Microsoft 스택 × Agent 365 역할

🔑 한 판 요약: 에이전트 액션은 Entra Agent ID(신원) → Prompt Shields·Purview(정책) → Defender Security for AI(통제) 게이트를 지나 실행되고, 전 과정이 Agent 365 레지스트리·DSPM·Defender XDR·Entra 인벤토리로 관측됩니다. 네 축을 Entra·Purview·Defender·M365 관리자 4개 역할이 나눠 맡고, 온보딩·수명주기 거버넌스가 전 과정을 관통합니다.
